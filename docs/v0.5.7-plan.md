# v0.5.7 plan

Two tracks: finish the three v0.5.6 gates that were deferred, and add
capability-driven panels so a file only shows the sections it can actually
support. Scope for the panel work is E57 + mesh profiles only; the LAS/LAZ and
COPC/EPT survey path stays exactly as it ships in v0.5.6.

## Track A — carried-over gates

### Gate 2 — per-cloud elevation filtering

Full staged design already written in `docs/gate2-per-cloud-filter-plan.md`.
Per-material elevation uniforms plus per-cloud pick windows, four stages each
with a `build:live` device check. Closing this lets the *Known limitations* note
come out of the changelog and release notes.

### Gate 5 — deterministic export frontier

The streaming resident snapshot (`snapshotResidentCloud` /
`residentSnapshot.ts`) concatenates every resident mesh. During a cross-fade the
outgoing parent node and its incoming children are both resident, so the export
can carry overlapping LOD samples for the same area.

Fix: before building the snapshot, compute a deterministic leaf frontier. Walk
the resident node set by octree key; for each path keep the deepest resident
node and drop any ancestor whose descendants are already resident, and exclude
nodes flagged as fading out. Put the frontier selection in a pure, tested module
that takes resident node keys + fade state and returns the keep set, so it is
verifiable without the GPU. The snapshot then concatenates only the frontier.

### Gate 10 — test coverage for the v0.5.6 GPU/catalog behaviour

The browser suite is green; what is missing is unit coverage for logic that
currently lives inside the `Viewer` (which constructs the real `WebGPURenderer`
and so cannot be instantiated in vitest).

- Extract the GPU-error dedup/cap/reset into a pure `gpuErrorLedger` (add,
  reset, cap) and test it directly. `Viewer` keeps a ledger instance.
- Test the catalog status transitions (`_setStatus` opening/error/info,
  `markLoaded`, `showOpenError`) in the ui bucket (CatalogPanel is DOM-only,
  runs under jsdom).
- For `uncapturederror` forwarding and `device.lost`, add a small injectable
  seam so a fake device object can drive `onGpuError` in a test, rather than
  needing a real GPU.

## Track B — capability-driven panels (E57 + mesh)

### The model

At load, derive a capability descriptor from `(sourceFormat, decoded
attributes, shape verdict)`:

- `hasRgb`, `hasIntensity`, `hasClassification`, `hasNormals`, `hasGpsTime`
  (most already exposed via `availableColorModes()` / `hasIntensity` /
  `hasClassification`);
- `isGeoreferenced` (a known CRS);
- `isMesh` (from `sourceFormat`: obj, glb, gltf, and ply when it carries faces).

Map the descriptor to a display profile. v0.5.7 adds two new profiles and leaves
the existing survey path as the default:

| Profile | Formats | Show | Hide |
|---|---|---|---|
| geo (unchanged) | LAS/LAZ, COPC/EPT | everything today | — |
| terrestrial-scan (new) | E57, PTX, PTS | point count + density, RGB / intensity / normals colour, bounds, per-point readout, measure, clip, section/profile | CRS + datum, classification, USGS Quality Level, ground coverage, DTM + contour readiness, vertical-accuracy-vs-datum |
| handheld-scan (new) | iPhone/iPad LiDAR (PLY, OBJ, GLB/GLTF, Y-up, small extent) | RGB / normals colour, real-world scale + bounds, measure, clip, per-point readout; a "handheld LiDAR scan" headline | intensity, classification, CRS + datum, USGS QL, ground coverage, DTM/contours, survey-density framing |
| mesh (new) | OBJ, GLB/GLTF, PLY-mesh (not identified as a handheld scan) | geometry, RGB / normals colour, measure, clip | intensity, classification, CRS, LiDAR density framing, all terrain sections |

Everything not in `terrestrial-scan`, `handheld-scan`, or `mesh` keeps the
current behaviour, so the survey path is untouched.

### Auto-identifying iPhone/iPad LiDAR scans

A handheld scan overlaps the mesh profile, so detection needs a reliable signal
rather than format alone. Candidate signals, strongest first:

- **Generator metadata.** glTF/GLB carries `asset.generator`; PLY and OBJ carry
  header comments. Apps stamp these (Polycam, Scaniverse, 3d Scanner App,
  RealityKit / RoomPlan, Apple). A case-insensitive match on a known set is the
  most reliable tell. The exact strings will be read off the sample file the
  maintainer is providing, so the match set is grounded in a real export, not a
  guess.
- **Geometry shape.** Y-up (already how `isZUpFormat` treats these formats),
  small metric extent (a room or object, single-digit to low-tens of metres),
  RGB present, no intensity, no classification, no CRS.

Detection returns a confidence, and the profile is applied automatically only on
a strong signal (generator match, or geometry heuristics agreeing); otherwise the
scan stays in the plain `mesh`/`terrestrial-scan` profile. A manual override
("treat as …") mirrors the existing terrain/object override so the user can
correct a miss. Detection logic goes in the same pure `displayProfile.ts` (or a
sibling `scanSourceDetect.ts`) and is unit-tested against fixture metadata.

### Implementation

- New pure module `src/render/displayProfile.ts`: `profileFor(descriptor)` plus
  a `sectionVisible(profile, sectionId)` table. Pure data in, enum + booleans
  out, fully unit-tested. Keeping it out of `AnalysePanel.ts` also avoids adding
  weight to an already-large file and sets up the F2 decomposition.
- `AnalysePanel` and the Inspector cards read `sectionVisible(...)` to decide
  what to render. The terrestrial-scan headline replaces "not placed on Earth"
  (which reads as a defect) with "Terrestrial laser scan — local coordinates",
  stated as a fact.
- The existing shape-based terrain/object routing stays; the profile gates on
  top of it (an E57 detected as non-terrain already avoids the terrain panel;
  the profile additionally removes the CRS/classification/QL rows the format
  can't support).

### E57 example

Replace the `tests/bunnyFloat.e57` fixture (a Stanford-bunny scan, not a real
terrestrial capture) with a representative terrestrial E57 that carries RGB +
intensity, so the new panel is exercised on real content.

Chosen source: the libE57.org test data (`http://www.libe57.org/data.html`),
whose licence permits free use, reproduction, and distribution, requiring the
copyright notice only in source (not binary) copies. Candidates, both a pump-room
terrestrial scan © 2008 Carnahan-Proctor and Cross, Inc.:

- `PumpARowColumnIndexNoInvalidPoints.e57` — XYZ + intensity + RGB, gridded,
  2.6 MB. Smaller; preferred as the repo test fixture.
- `PumpACartesian.e57` — XYZ + intensity + RGB, 5.3 MB. The canonical PTS-like
  scan; a good in-app sample if we add one.

Add the © notice to `THIRD_PARTY_NOTICES.md`. Optionally add a first entry to the
empty `SAMPLES` array so the empty state offers a "try an E57" button.

For the handheld profile, the maintainer is providing a well-formed iPhone/iPad
LiDAR sample; its generator metadata is read to build the detection match set and
it becomes the handheld-scan test fixture. Its licence/attribution also go in
`THIRD_PARTY_NOTICES.md`.
