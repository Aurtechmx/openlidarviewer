### Gate 5 â€” deterministic export frontier

The streaming resident snapshot (`snapshotResidentCloud` /
`residentSnapshot.ts`) concatenates every resident mesh. During a cross-fade the
outgoing parent node and its incoming children are both resident, so the export
can carry overlapping LOD samples for the same area.

Fix: before building the snapshot, compute a deterministic leaf frontier. Walk
the resident node set by octree key; for each path keep the deepest resident
node and drop any ancestor whose descendants are already resident, and exclude
nodes flagged as fading out. Put the frontier selection in a pure, tested module
that takes resident node keys + fade state and returns the keep set, so it is
verifiable without the GPU. The snapshot then concatenates only the frontier.

### Auto-identifying handheld / object captures

A handheld capture overlaps the mesh profile, so detection needs a signal beyond
format alone. The maintainer's real samples (`Statue1.glb`, `Statue2.glb`) set
the ground truth here, and they force a more honest design than the first draft
assumed:

- **Generator metadata is often absent.** Both sample GLBs carry no
  `asset.generator`, no `copyright`, no `extras`, no `extensionsUsed`, and no
  node names â€” the export is stripped clean. So generator matching cannot be the
  primary signal. It stays as a *high-confidence bonus* when present (glTF
  `asset.generator`; PLY/OBJ header comments; apps like Polycam, Scaniverse, 3d
  Scanner App, RealityKit / RoomPlan), but the common case has nothing to match.
- **Structure + geometry is the primary signal.** The reliable fingerprint of
  these captures: a single mesh / single unnamed node, indexed `TRIANGLES`, a
  baked-texture PBR material (`baseColorTexture`, `metallicFactor: 0`, embedded
  JPEG, hex-string image names), colour via `TEXCOORD_0` rather than `COLOR_0`,
  object/room-scale metric extent (sub-metre to low-tens of metres), and no
  intensity / classification / CRS. Vertex RGB or normals may be present in other
  exports but are *not* in these â€” so the colour branch must cover textured
  materials too.
- **Up-axis is not assumable.** In both samples the vertical (Y) axis is the
  *smallest* extent and there is no root rotation node, so the object is not
  cleanly Y-up-standing. Detection must NOT auto-rotate on a Y-up assumption
  (it would tip the model over); instead the profile exposes a manual up-axis
  flip.

Detection returns a confidence tier. A generator match is high confidence and
names the source; the structure/geometry fingerprint alone is modest confidence
and yields the neutral "handheld / object capture" headline **without** claiming
a specific device (the file does not prove "iPhone LiDAR", so we do not assert
it). The profile is applied automatically only on a strong-enough signal;
otherwise the scan stays in the plain `mesh` profile. A manual override
("treat as â€¦") mirrors the existing terrain/object override so the user can
correct a miss. Detection logic goes in the same pure `displayProfile.ts` (or a
sibling `scanSourceDetect.ts`) and is unit-tested against synthetic descriptors
that reproduce the sample fingerprint (single textured mesh, no generator,
object-scale extent), not against the binaries.

### Implementation

- `AnalysePanel` and the Inspector cards read `sectionVisible(...)` to decide
  what to render. The terrestrial-scan headline replaces "not placed on Earth"

  the profile additionally removes the CRS/classification/QL rows the format

New feature this unlocks â€” **surface `olv:` provenance metadata.** The Tikal
sample embeds the `https://aurtech.mx/openlidarviewer/metadata/1.0` namespace
with fields that match this project's honesty stance: `creator`,
`organization`, `license`, `publicationStatus` (`research_sample`),
`accuracyClass` (`reference_based_not_survey_grade`), an explicit `limitations`
string ("not photogrammetric, not archaeological survey data, not suitable for
conservation measurements or survey certification"), `sourceBasis`,
`datasetType`, `totalHeightMetres`, `levelCount`. The terrestrial-scan profile
should read this namespace when present and render a provenance/honesty card,
and a declared `accuracyClass=reference_based_not_survey_grade` should suppress
any survey-grade framing outright. El Elegante (no `olv:` block) exercises the
absence path. The `coordinateMetadata` string ("X=north-south across facade â€¦
Z=up â€¦ no geodetic CRS") can supply the local-frame description in the headline.

Reader correctness â€” **E57 CRC paging.** These pye57 files use the standard
paged E57 layout (1024-byte pages = 1020 data + 4-byte CRC-32) for both the XML
header and the binary point records. The reader must strip/validate CRC per
page; the libE57 pump file loading does not prove these parse, so both get an
on-device load test. CRC validation also lets us fail honestly on a genuinely
corrupt file rather than mis-reading it.

Sizes are large (28.5 MB + 36.4 MB E57, 47 MB CSV), so these stay local
verification samples, not committed binaries; CI uses synthetic descriptors.
The Tikal E57 is the strongest in-app `SAMPLES` demo candidate (the honest-
provenance showcase), likely decimated, if we add an empty-state "try an E57"
entry.

### Test data strategy

Two sources, on purpose:

- **Real examples** for realism and the on-device panel checks: the libE57
  pump-room E57 (terrestrial-scan) and the maintainer's iPhone/iPad capture
  (handheld-scan). Both are attributed in `THIRD_PARTY_NOTICES.md`.
- **Our own generated synthetic fixtures** for deterministic CI: profile and
  detection logic is a pure function of a capability descriptor plus metadata,
  so the unit tests build synthetic descriptors and synthetic
  generator/comment strings (extending the existing `tests/fixtures/**` and
  `synthCopc.ts` pattern). CI coverage never depends on a downloaded file; the
  real examples are for human-facing verification and the demo sample.
