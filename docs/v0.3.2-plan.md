The agreed scope for v0.3.2, captured in-tree so every commit can be
checked against it. v0.3.2 closes out the work deferred from v0.3.1 and
prepares the platform seams that v0.3.3 will build on. It is **not a
feature release** — it ships zero user-visible UI changes by default.

## Semver-patch contract

Everything that worked in v0.3.1 must keep working: every supported format,
COPC streaming (local + remote), measurement, annotation, point inspection,
the live probe, saved views, session round-trip (JSON v2), share links, the
WebGPU primary path, the WebGL2 fallback, and both the plain and the
obfuscated builds.

## Non-goals

- No new file formats (EPT is staged in v0.3.3).
- No new image-export modes (the heatmap / intensity / classification /
  depth exports are staged in v0.3.3 — v0.3.2 lands the seam only).
- No new measurement tools (cross-section / clip planes — later).
- No breaking changes to sessions, share links, the embed API, or the
  on-disk JSON v2 shape.
- No SDK surface changes beyond what the streaming-source refactor needs.

## What v0.3.3 needs from v0.3.2 (the seam contract)

v0.3.2 succeeds when the v0.3.3 work can plug in without touching the
scheduler, the renderer, or the session schema. Specifically:

1. **EPT (Entwine Point Tile) support** — v0.3.3 must be able to add an
   `EptStreamingSource` class that conforms to a `StreamingSource`
   interface, and `attachStreamingCloud` must accept either source kind
   with no scheduler / renderer change.
2. **Advanced image exports** (orthographic, heatmap, intensity,
   classification, depth) — v0.3.3 must be able to add new render modes
   to an `ImageExportRegistry` without modifying the existing screenshot
   path.

These two seams are the central deliverable of v0.3.2.

## Phase 1 — Three.js deprecation cleanup

Acceptance: zero deprecation warnings in the production console at scan
open. The two warnings observed in the live v0.3.1 build:

- `THREE.Clock: This module has been deprecated. Please use THREE.Timer
  instead.` — one callsite: `Viewer._clock = new THREE.Clock()`.
- `THREE.PostProcessing: "PostProcessing" has been renamed to
  "RenderPipeline".` — two callsites: the type and the `new`.

Pure refactor. No behavior change, no API change. Tests + builds stay
green.

## Phase 2 — Incremental rescoring (deferred Task 10 from v0.3.1)

Acceptance: mean scheduler tick ≤ 2 ms on the 100 M synthetic fixture
(the v0.3.1 plan's acceptance, which the existing full-rescore loop misses
beyond ~10 M nodes); tick time scales with the dirty-set size, not with
total node count; the post-stop "no late onNodeReady" invariant from
v0.3.1 still holds.

The current scheduler walks every node in the cloud every tick. The
target: walk only nodes whose visibility, depth-relative-to-cap, or
projected-size has changed since the last tick. The unchanged nodes keep
their prior score.

- **Task A — Dirty set.** Maintain a `Set<NodeId>` of nodes whose
  inputs to `nodeScore` may have changed since the last tick. A frustum
  shift, a camera-velocity change, or a budget change marks new nodes
  dirty. A node that just entered or just left the frustum is dirty by
  definition.
- **Task B — Frame-budgeted re-score.** Bounded N-per-tick (default
  256) rescores. Excess dirty nodes carry to the next tick. The
  acceptance metric is the tick-time histogram, which the existing
  streamingBenchmark already records.
- **Task C — Selection over the dirty + stable set.** `selectWithinBudget`
  still consumes a sorted score list; that list is now a merge of the
  fresh-scored dirty subset with the cached score of the stable subset.
- **Task D — Unit tests.** Camera at rest → dirty set empties after one
  tick; sustained pan → dirty set tracks the frustum's leading edge;
  budget change → every node is dirty for one tick.

The eviction state machine, the cache hysteresis, and the pressure
adaptation from v0.3.1 are untouched.

## Phase 3 — Streaming source abstraction (v0.3.3 platform prep)

Acceptance: a hypothetical `EptStreamingSource` could be added to
v0.3.3 by implementing one interface, with zero changes to the
scheduler, the renderer, or the picking path.

Extract the common interface that COPC currently satisfies implicitly.
The existing pieces:

- `StreamingPointCloud` — opens a `RangeSource`, parses COPC metadata,
  exposes octree + node records + render origin.
- `StreamingScheduler` — consumes a `StreamingPointCloud` to do the
  view-dependent work.
- `StreamingRenderer` — consumes the scheduler's `onNodeReady` /
  `onNodeEvicted` callbacks plus the cloud's render origin.

The needed interface — call it `StreamingSource` — exposes:

- `kind: 'copc' | 'ept'`
- `octree: ReadonlyOctree` (the existing `StreamingNodeStore` view)
- `metadata: { name, sourceFormat, header, info, sourcePointCount }`
- `renderOrigin: [number, number, number]`
- `residentPointCount`, `counts()` (already on `StreamingPointCloud`)
- `readNodeChunk(record, signal)` (already on `StreamingPointCloud`)

The refactor: rename `StreamingPointCloud` to `CopcStreamingSource`,
extract the interface above (`StreamingSource`), wire the scheduler /
renderer / Viewer to depend on the interface, leave the old class name
as a type alias for one release for backward compatibility on any
downstream consumer.

Pure refactor. No behavior change; no UI change; the existing tests
must pass unchanged against the renamed implementation.

## Phase 4 — Image export framework seam (v0.3.3 platform prep)

Acceptance: a hypothetical `ImageExportMode = 'heatmap'` could be added
to v0.3.3 by implementing one factory function and registering it,
without touching the Viewer.

v0.3.2 lands:

1. **`ImageExportRegistry`** — a name → factory map of `ImageExportMode`
   values. v0.3.2 registers exactly one mode: `'orthographic'`, which
   renders the current scene through a temporary `OrthographicCamera`
   matching the active perspective view's framing, captures to PNG via
   the existing `canvas.toBlob` path.
2. **`renderOffscreenImage(mode, options)`** — Viewer method that the
   registry calls. Pure orchestration; the mode factory does the work.
3. **An "Export image" button group** in the Inspector that, in v0.3.2,
   exposes only "Orthographic PNG". v0.3.3 fills in the other four.

The existing PLY / OBJ / XYZ / CSV cloud-data exports are untouched.
The existing "Save view" (Cmd-S) screenshot is untouched — both routes
continue to work via the same `toBlob` capture.

The orthographic mode itself is the simplest image-export. Its presence
in v0.3.2 proves the seam works and gives the v0.3.3 modes a concrete
reference shape to copy.

## Definition of done

v0.3.2 ships when, on the live build:

1. Both Three.js deprecation warnings are gone from the production console.
2. Mean scheduler tick ≤ 2 ms on the 100 M synthetic fixture, with the
   incremental-rescoring path enabled.
3. The v0.3.1 e2e suite (URL field, keyboard orbit, chunks-load smoke
   test, the 1 M stress harness) stays green on the obfuscated build.
4. An orthographic PNG export round-trips through the new
   `ImageExportRegistry` with byte-equal output to the existing
   "Save view" path on the same view.
5. The renamed `StreamingSource` interface is exported and a stub
   `EptStreamingSource` test class can be instantiated without runtime
   error (it never decodes — proves the interface alone is enough).

## Quality bar

Production-grade. Typed. Frame-budget-aware. No optimisations that aren't
measured. No hacks that survive a code-review pass. No "while I'm here"
refactors that aren't on this list.

## Out of scope for v0.3.2 (queued for v0.3.3)

- The four image-export modes beyond orthographic: heatmap, intensity,
  classification, depth.
- EPT format support (the `StreamingSource` interface is here; the
  EPT implementation is not).
- SDK / embed-API hardening.
- The 50-scan open/close leak audit and the WebGL2-forced streaming
  e2e (the live-browser QA passes deferred from v0.3.1).
