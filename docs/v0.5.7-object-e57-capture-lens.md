# v0.5.7 — Object / E57 capture lens

## Problem

The app characterises the same scan two contradictory ways. A local E57 of
Tikal Temple I:

- the shape router classifies it **Object** (compact 3-D, non-terrain) and the
  left panel correctly says "terrain analysis does not apply";
- the capture-type classifier, blind to that verdict, labels it **"Drone-mounted
  LiDAR (UAV ALS) — medium confidence"** purely from density
  (572 pts/m² over a 0.27 ha footprint);
- the Inspector's **Technical Report** then renders the object as airborne survey
  data — mapping-footprint coverage, GSD planimetric/elevation tolerances,
  NVA = 1.96 × RMSEz, "ground/feature extraction needs classification",
  "exports cannot be georeferenced";
- meanwhile the ObjectPanel's **space report** describes it correctly as an
  object.

So a client (or a researcher citing the data) receives two PDFs: one says
"Object", the other says "drone survey". For a tool whose purpose is **valid,
research-usable data**, recording a temple as aerial drone LiDAR is a provenance
error, not just clutter.

Root cause: three surfaces — the capture-type classifier, the report entry
points, and the right-rail panel sections — each decide **independently** whether
"survey / terrain framing applies", and they disagree.

## Goal

One coherent **capture lens** so an object / local-frame scan is consistently
characterised, with survey/terrain framing hidden or corrected everywhere.
Priority is data validity for research use: the classifier honesty fix is
load-bearing; the panel/report decluttering follows from the same predicate.

## The capture lens (single source of truth)

A pure value computed once at load, carrying the two verdicts the app already
produces plus a derived flag:

```ts
interface CaptureLens {
  readonly isNonTerrain: boolean;   // scanShape: object | interior
  readonly isLocalFrame: boolean;   // displayProfile: terrestrial-scan | handheld | mesh (no geodetic CRS)
  readonly isCaptureScan: boolean;  // isNonTerrain || isLocalFrame
}
```

It carries the **facets**, not just the boolean, because the two facets gate
different wrongness and must not be collapsed:

- **Survey / CRS / aerial framing** is wrong for anything **local-frame**
  (driven by profile). Applies regardless of shape.
- **Terrain framing** (contours, DTM, ground coverage, slope, terrain
  complexity) is wrong for **objects / interiors** (driven by shape).

A local terrestrial scan of a *hillside* is terrain-shaped but has no CRS: it
keeps terrain contours but drops CRS/survey framing. One collapsed boolean would
mis-handle it; the shared facet object handles it and prevents the surfaces from
re-deriving the answer independently (the current bug).

Lives in a small pure module (`src/render/captureLens.ts`), unit-tested from
synthetic `(shapeVerdict, profile)` inputs. Consumed via the existing lazy
`applyDisplayProfile` chunk so the tight eager `index` budget (769/772 KiB) is
untouched.

## Surface 1 — capture-type classifier (the honesty fix)

`src/diagnostics/provenance.ts` `classify(signals)` is shape-blind;
`matchNumeric` infers drone/aerial from density. Change:

- Thread the shape verdict (`isNonTerrain`) into `ScanSignals` (or a sibling
  param).
- When `isNonTerrain`, the classifier **may never** return an aerial type
  (`aerial`, `drone`/UAV ALS, `spaceborne`). It returns a terrestrial / object
  characterisation, or defers ("Object scan — capture method not determined") at
  appropriate confidence. The density-only aerial heuristic is suppressed.
- When `isLocalFrame`, survey framing ("survey-grade", CRS-dependent language)
  is not asserted.
- Explicit user overrides (`fingerprintFor`) and strong software/sensor matches
  (a Polycam export is iPhone-LiDAR) still win — those are direct evidence, not
  density guesses.

This fingerprint feeds both the Provenance card and any report's capture-type
line, so fixing it here fixes it everywhere downstream. Pure and unit-tested: a
`nonTerrain` input must never yield an aerial capture type.

## Surface 2 — reports ("object report only")

- The Inspector's terrain/technical Report PDF (and its template picker) is
  offered only for **terrain-shaped** scans. Objects / interiors get only the
  ObjectPanel's object/space report — no contradictory technical document.
- Inside any report that IS shown (a terrain report for a terrain scan), the
  survey/aerial content (CRS, GSD, NVA, aerial capture type) is suppressed when
  `isLocalFrame`.
- Net for the temple: one report, the correct object one.

## Surface 3 — panel declutter (gated by facet)

- `isNonTerrain` hides the terrain-only right-rail surfaces: the
  Coverage / Confidence colour modes (and their "run terrain analysis first"
  note, which never resolves for an object), the terrain Dataset-Intelligence
  rows (Terrain Complexity, Ground Visibility, and the dashed Metric Stability
  placeholder), the terrain workflow presets, and the Inspector Scan Report /
  Report PDF sections.
- `isLocalFrame` hides the Coordinate System section (the existing v0.5.7 CRS
  gate — **confirm it actually fires for the E57**; a device screenshot suggests
  it may not) and any survey-grade framing.
- Exact section list finalised during a code pass; classification legend and
  intensity are already gated by attribute presence.

## Verification

- Pure `captureLens` + the classifier's shape-awareness: unit tests (CI). The
  key invariant test: `isNonTerrain ⇒ captureType is never aerial`.
- Report-button gating and panel gating are DOM/shape-driven and device-visual;
  they land with on-device confirmation, and must not break existing e2e/unit
  assertions (`v036Features`, `analysePanel`, `scanRoute` specs).
- The CRS-hide "is it firing on the E57" question is run down as part of
  surface 3.

## Out of scope (YAGNI)

- No new report templates; no merging of the two report buttons (suppression was
  chosen, not merge).
- No change to the survey/terrain path itself — a georeferenced airborne LAS is
  unaffected.

## Order

1. `captureLens.ts` pure predicate + tests.
2. Classifier shape-awareness + tests (the data-validity fix).
3. Report-button gating (object report only) — device-visual.
4. Panel declutter via the lens + CRS-hide confirmation — device-visual.
