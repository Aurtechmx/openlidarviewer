# v0.6.0-alpha.1 — pre-roadmap hardening report

**Scope.** A numerical, lifecycle, memory, and reproducibility pass over the
`v0.6.0-alpha.1` branch before the Integrated Spatial Workflows roadmap is built
on top of it. This report records what was checked, what was found, what was
fixed with test evidence, what remains, and a single readiness verdict.

**Artifact.** Branch `v0.6.0-alpha.1`. First written at HEAD `1c316ce`; revised
after that commit, so the fix set below now runs past it — the release manifest
in each archive pins the exact commit that archive was cut from, and is the
authority on what a given zip contains. Working tree clean at package time
(`release-manifest` records `dirtyWorkingTree: false`).

**Evidence rule.** Every "Passed" below is backed by a command run against this
commit. Checks that need a browser, a GPU, or a heap profiler are marked **Not
executable here** — they are real gaps in this report's coverage, not passes.
No measurement in this document is estimated or invented.

## Correction — this report missed a measurement defect

The first version of this document reported the measurement surface as checked
and named no defect in it. That was wrong, and the way it was wrong is worth
recording, because the same blind spot could be repeated.

Every profile measurement printed a render-LOCAL height labelled as an
elevation. On a streaming COPC whose render origin sits ~830 m up, the panel
showed a surveyor **−481 m for ground that is at +349 m** — in the chart, the
summary, the CSV and the PDF alike. It was found by reading a screenshot against
the source file's LAS header (Z 330.030..467.150), not by any check listed below.

Two reasons the suite here could not see it, both instructive:

  * Every test compared the measurement stack against *itself*. The heights were
    self-consistent end to end, so nothing disagreed. The defect was only visible
    against a fact from outside the code — the file's own header.
  * The error is a constant offset, and a constant offset cancels in a
    subtraction. Every delta the suite asserted — gain, loss, grade, steepest
    section — was correct, and stayed correct. Only the absolutes were wrong,
    and nothing asserted an absolute against an external truth.

The defect predates this report and shipped in the archives cut before it (see
D5). It is fixed and tested now. The lesson generalises past this bug: a
self-consistent measurement stack proves nothing about its datum, and this
report's "Passed" rows for the measurement surface carried more confidence than
their evidence supported.

## Confirmed defects — fixed with test evidence

Each was reproduced against this source before any change, then fixed test-first
(a failing test that goes green with the fix).

### D1 — Session origin rebase was incomplete

`rebaseSessionGeometry` shifted measurement and annotation vertices into the
imported cloud's frame but left the live camera, saved-view cameras, the global
clip box, per-view clips, profile-chart elevations, volume reference planes, and
annotation jump-to-view cameras in the original frame. On a cross-origin import
the geometry landed correctly while every camera and clip pointed at the wrong
place.

**Fix.** The transform now shifts every local spatial field by the same delta,
with elevation-only scalars (profile heights, volume `referenceZ`) moved by the
up-axis component, and the importer applies the rebased views/camera/clip.
**Tests:** camera, saved views, per-view clip, profile elevations, volume
reference, annotation camera, Y-up elevation delta, and a values-preserving
zero-delta no-op.

**Deliberately left in the source frame:** the elevation point-filter window,
whose stored units are ambiguous between the source and local frames. Shifting
it risked introducing a wrong transform; it is called out here rather than
guessed at.

### D2 — Non-finite points survived voxel downsampling

`PointCloud.bounds()` ignores non-finite coordinates for the camera, but
`voxelDownsample` did not: a NaN or infinite coordinate fell into its own voxel
key, summed to a non-finite centroid, and rode through into the reduced cloud
and any later analysis.

**Fix.** The voxel loop skips a point whose x/y/z is not finite. **Test:** a
cloud mixing NaN, +Inf, and −Inf inputs yields only finite output points.

That guard alone was narrower than the finding deserved, and the gap is now
closed at the source: one shared policy (`src/io/sanitizeCloud.ts`) that all
eight file loaders route through. It drops an unplaceable record together with
its parallel colour / class / intensity / return / GPS-time entries in lockstep
— keeping a parallel array at full length while positions shrink would
mis-align every attribute after the dropped point, which is a worse bug than the
NaN itself — counts the exclusions, reports them through the existing
`loadWarnings` channel, and refuses a file with nothing placeable left rather
than loading it empty. The filter runs BEFORE the origin is computed:
`floor(min)` over a cloud containing −Infinity is −Infinity, and subtracting
that origin would turn every survivor into NaN. Streaming nodes and
`voxelDownsample` output stay out of this path by design.

`loadGltf` was the last loader outside the policy and is now routed through it.
The gap was real rather than theoretical: a glTF whose POSITION buffer holds a
NaN loaded three points where two are placeable, and the bad vertex reached the
cloud. glTF carries no georeferencing, so it keeps its zero origin and only the
points and their colours are filtered.

### D3 — Decode requests could leak on a synchronous post failure

In the EPT and COPC worker clients a request was registered in the pending map
before `postMessage`, and the post was not guarded. A synchronous failure (a
detached transfer buffer, a terminated worker) rejected the caller's promise but
left the request in the map with its abort listener still attached — one leaked
entry and one leaked listener per failure. `_failAll` also cleared the map
without detaching abort listeners.

**Fix.** A single settle path removes the map entry and detaches the listener;
it is used by every reply, by a guarded post that rejects on a synchronous
throw, and by `_failAll`. **Tests (both clients):** a synchronous post failure
rejects, leaves an empty pending map, detaches the listener, and does not wedge
the client; dispose detaches listeners so a later abort is inert.

### D4 — Stockpile density confidence used native units against a metric bar

The high-confidence grade compares sample density against 5 points/m², but was
fed the raw native figure (points per horizontal-unit squared). On a foot-unit
survey that read roughly 11× too low and downgraded genuinely dense stockpiles.
The displayed density was already unit-correct — only the internal grade
threshold was affected. This is **narrower than the external review stated**
(the review implied the displayed figure was also wrong; it was not).

**Fix.** Density is converted to points/m² with the CRS linear-unit factor
before grading; metre projects are unchanged. **Test:** a foot-unit survey at
~1.2 pts/ft² (~13 pts/m²) grades high, while the identical native density read
as metres stays below high — the bar is unit-aware, not removed.

### D5 — Profile elevations were printed against the wrong datum

Profile heights are sampled and stored in render-local space, and nothing
restored the source datum before printing them. `MeasureController.setContext`
received the scan origin and, by its own comment, held it "reserved for later
phases" — so every printed profile elevation was short by the origin's up-axis
component. See the correction above for the measured example.

**Fix.** The origin's up-axis component is added at the display/export boundary
only; storage stays local, because the session importer rebases stored heights
by the up-axis delta and would break if they became absolute. A second defect
travelled with it: `formatDistance` chose its unit with `meters < 1` rather than
`Math.abs(meters) < 1`, so any negative value rendered as centimetres
(`-41186.5 cm`) — the imperial branch already used `Math.abs`.

A scene has one datum or none. Because two clouds with different origins are
already drawn in frames that do not line up, and a measurement's vertices do not
record which cloud they were taken against, picking one cloud's origin would
hand its frame to points that were never in it. So the datum is resolved by
unanimity (`resolveSceneOrigin`), the same gate the georeference seam already
applies, and a conflict REFUSES the datum rather than guessing: the columns
rename to "Local height" and a notice rides the panel, CSV and PDF. The refusal
repaints on its own rather than depending on an unrelated broadcast.

### D6 — A parse-worker post failure could hang an unrelated load

Both worker paths in `loadFile.ts` registered an abort listener before an
unguarded `postMessage`. A synchronous serialisation failure (`DataCloneError`)
rejected the caller but skipped `detach()`, leaving the listener attached with
`settled` false and holding a live reference to the SHARED parse worker. A later
abort on that signal then nulled the handlers of, and terminated, the worker a
DIFFERENT healthy load was decoding on — that load never settled, and because it
never released the worker gate, every load queued behind it stopped too.

**Fix.** Both posts are guarded and settle through the existing teardown. The
worker is deliberately NOT dropped: a post to a terminated worker discards
silently rather than throwing, so a synchronous throw always means serialisation
failed and the worker never saw the message — it is idle and healthy, and
dropping it would discard the warm decoder the shared worker exists to keep.

## Numerical hardening

A shared compensated-statistics module was added (Welford streaming variance,
Kahan–Neumaier summation) and applied where the one-pass forms were unsafe.

- **Stockpile thickness variance** used `Σt²/n − mean²`, which loses the entire
  variance to cancellation when the spread is tiny next to the mean (a tall pile
  with a smooth top). Now streamed through Welford. A foil test confirms the
  naive form collapses to ~0 at a 1e8 offset where Welford recovers the true
  0.5 variance.
- **Cross-check RMSE** and **held-out RMSE** aggregates moved onto the
  compensated sum so they do not drift over a large grid.

Values are unchanged on ordinary inputs; the guards matter only at the offsets
and sample counts real scans reach. **Tests:** order-independence, the
large-offset cancellation foil, a million-term summation-drift comparison, and
known-value checks.

**Not converted (lower risk, positive-term sums over smaller N):** the per-band
and per-zone RMSE sub-accumulators, ICP residual sums, and terrain variance
paths. They should adopt the same module before more scientific workflows depend
on them, but none is a cancellation case.

## Areas checked but not changed

- **Coordinate precision on load.** The float64 recenter-before-narrowing path
  for PCD/PLY/OBJ holds. Known residual limits: binary-PLY double coordinates
  and de-indexed OBJ faces still depend on loader-provided float32 positions;
  large ASCII bodies parse via `split`, which is memory-heavy. Documented, not
  fixed here.
- **Worker death recovery.** COPC and EPT clients mark themselves broken after a
  fatal worker error and reject subsequent requests instead of hanging.
  Confirmed by test.
- **Deterministic streaming terrain samples.** Resident nodes are sorted by
  stable octree key before subsampling, so a fixed resident set yields a fixed
  sample. The resident set itself remains non-deterministic (camera, budget,
  load order) — by design, and disclosed.

## Remaining blockers before the roadmap

1. **Connect the streaming pipeline controls** (GPU upload queue, pipeline
   limits, adaptive tiers) to the runtime. Browser-verifiable only, and the
   single largest thing standing between this branch and the roadmap.
2. **Converter cancellation and output memory** (preflight estimate, visible
   cancel, release of downloaded output).
3. **Reprojection to a projected working CRS**, offered in a worker. This is the
   remaining part of the geographic-CRS policy, and it is a feature rather than
   a hardening fix — the honesty half is already in place and was wrongly listed
   here as outstanding. Lon/lat analysis is already exploratory-only: the
   Contour Studio refuses a validated deliverable while the CRS is geographic
   (`contourStudioLaunchState`), epoch alignment refuses a geographic frame
   outright, and Measure / Analyse / Inspector each carry a standing notice that
   degrees are not metres and that lengths, areas, grades and profiles are not
   reliable distances until reprojected. What the app cannot do is reproject FOR
   the user, so the honest path out of a geographic frame is currently to bring
   a projected file.
4. **Duplicate streaming node keys** are still ordered by arrival among
   themselves, so the sampler's determinism guarantee assumes unique keys.

Cleared since the first version of this report: the numerical adoption is
finished (per-band RMSE, ICP, spatial-block), the ASCII loaders are scanned
rather than split whole (PLY, PCD; PTX indexes line offsets instead of holding
a string per line), and the terrain sampler's contract is guarded — including a
real defect that pass found, where a short class array wrote `undefined` into a
`Uint8Array` and silently became class 0 ("never classified") in place of the
255 "no class channel" sentinel.

## Verdict

**NOT READY for v0.6 roadmap implementation.**

Six confirmed correctness and lifecycle defects (D1–D6) are fixed with tests,
the named variance-cancellation risk is hardened, and loader sanitation is
centralised. That makes this a materially stronger alpha than the reviewed
branch, and safe for continued internal alpha testing.

That count moved from four to six after this report first claimed the surface
was checked, which is the more useful signal in it: the two additions (D5, D6)
were found by reading real user output against a real file header, and by
tracing a lifecycle by hand — not by any check listed above. Read the pass rate
below as evidence about the paths that ARE tested, not as coverage of the
product.

It is **not** yet the foundation for building the universal analysis, project,
derived-layer, and Compare Studio systems, for two reasons that this environment
cannot close: the streaming pipeline controls are built but not connected, and
the browser/GPU/heap behaviour is unverified on this commit. Those, plus the
remaining numerical adoption, gate the move to "ready".

**Safe to begin now:** work that does not depend on the streaming pipeline or on
unverified GPU/heap behaviour — schema definition, the shared sampler and
analysis-runner interfaces, and further AppRuntime lifecycle extraction.

**Must stay blocked:** anything that drives heavier streaming or GPU load until
the pipeline controls are connected and verified in a real browser.
