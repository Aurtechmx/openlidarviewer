# v0.5.4 Terrain Science Hardening — Phase 0 audit

Scope: what already exists that the new terrain-complexity metrics (TPI per
Weiss 2001, VRM per Sappington et al. 2007) must REUSE, and the exact
conventions they must consume. No code changes in this phase.

Licensing note recorded up front: pyTopoComplexity is AGPL and MUST NOT be
read or ported. The new cores are implemented from the primary literature
only (Weiss 2001 poster; Sappington, Longshore & Thompson 2007, *J Wildl
Manage* 71(5), doi:10.2193/2005-723), cited in header comments.

## 1. The DTM grid, per-cell confidence, and hold-out RMSE structures

### 1.1 `DemRaster` — `src/terrain/ground/rasterizeDtm.ts`

The raw rasterisation product. Row-major `Float32Array z` with **`NaN` as the
honest "no data" state** (the rasteriser never invents data), parallel
`Uint32Array counts` (ground returns per cell), `cols/rows`,
`cellSizeM` (source linear units — NOT necessarily metres; see §2.3),
`originH1/originH2` (grid origin in the two horizontal axes), plus the honesty
tail: `coverage: TerrainCoverageMode`, `sourcePointCount`,
`analyzedPointCount`, `filledCellCount`, `warnings: string[]`.

Cell indexing: `col = floor((h1 − originH1)/cellSizeM)`,
`row = floor((h2 − originH2)/cellSizeM)`, index `i = row*cols + col`.
**Row+1 is one step in +h2 (northing) — see §2.2.**

### 1.2 `DtmGrid` — `src/terrain/ground/cellConfidence.ts`

The confidence-aware DTM (`buildDtmGrid(raster, params)`) — "the product;
contours, hillshade, exports, and the confidence overlay are all projections
of it". Fields the complexity metrics can consume directly:

- `z: Float32Array` — filled (never NaN) wherever `coverage > 0`; genuine
  gaps (`coverage === 0`) carry no trustworthy height.
- `confidence: Float32Array` — 0..100 per cell, derived from observable
  quantities only (density vs scene median + absolute half-count for measured
  cells; distance-to-data × Horn-roughness penalty × extrapolation guard for
  interpolated cells). Single source of truth for uncertainty — **no other
  module recomputes it, and the new metrics must not either.**
- `coverage: Uint8Array` — per-cell provenance `0 none / 1 interpolated /
  2 measured` (`CellCoverage`). This is the validity mask the complexity
  cores should honour (`coverage === 0` ⇒ skip as NoData).
- `counts`, `interpDistanceCells`, `cols`, `rows`, `cellSizeM`,
  `originH1/H2`, `crs`, `verticalDatum`.
- Whole-grid honesty: `coverageMode`, `sourcePointCount`,
  `analyzedPointCount`, `meanConfidence`, `warnings`.
- `gradeForConfidence()` + `EVIDENCE_THRESHOLDS` (solid ≥66 / dashed ≥33 /
  gap) — the shared visual grammar.
- `isHonestDtm(g)` — structural guard (lengths, confidence range, finite z
  where coverage > 0).

### 1.3 Hold-out RMSE — `src/terrain/validate/holdoutRmse.ts`

`holdoutValidateDtm` withholds a deterministic (seeded mulberry32) fraction of
ground returns, rebuilds the DTM, and measures vertical residuals at withheld
points, bucketed by evidence grade and by slope band (flat <5°, moderate <20°,
steep — thresholds stored as rise/run via `Math.tan`). It consumes the same
`hornSlope` + `horizontalCellMetresXY` as the live stage so validation and
display share one slope definition. Residuals are reported in metres via
`verticalUnitToMetres`. This is the template for how a new metric earns its
confidence claims: derived, seeded, falsifiable — never asserted.

## 2. Where slope + aspect already exist (VRM MUST consume these)

### 2.1 `hornSlopeAspect` — `src/terrain/ground/terrainDerivatives.ts`

Horn (1981) 3×3 weighted central difference. Units and conventions, exactly:

- **`slope` is rise/run — a dimensionless TANGENT** (`hypot(dzdx, dzdy)`),
  NOT radians and NOT degrees. `0` on flat ground. Consumers that need an
  angle convert themselves (`computeSlopeDegrees` in hillshade.ts does
  `atan(slope)·180/π`; holdoutRmse compares against `tan(threshold°)`).
- **`aspect` is in RADIANS, in the MATH frame: CCW from EAST (+x), with
  π/2 = north.** It is the **DOWNSLOPE** direction
  `atan2(−dz/dy, −dz/dx)` — i.e. the direction of −∇z, NOT the compass
  azimuth and NOT the upslope normal direction.
- **Sentinel:** `aspect = 0` where `dzdx === 0 && dzdy === 0` (flat cell),
  and `slope = 0, aspect = 0` where the centre cell is non-finite. Harmless
  for VRM: a flat cell's normal has zero horizontal component, so its
  (arbitrary) aspect never enters the resultant direction.

### 2.2 Northing-up grid convention (the v0.4.3 hillshade-mirror fix)

Our rasters are **NORTHING-UP**: `rasterizeDtm` maps
`row = floor((y − origin)/cell)`, so **row+1 is one cell NORTH** — the
opposite of image rasters. Hence `dz/dy` in `hornSlopeAspect` is
+∂z/∂northing and the downslope aspect negates BOTH gradient components
(`atan2(−dzdy, −dzdx)`). The ESRI textbook form `atan2(dzdy, −dzdx)` assumes
image rows (row+1 = south) and mirrored aspect north–south on our grids
(v0.4.3 bug: hillshade lit the wrong flank). `hillshade.ts` documents that
`azimuthToMathRad` converts the sun azimuth into this same math frame so
`cos(az − aspect)` compares like with like. **Any new consumer of `aspect`
must interpret it in this exact frame.**

- Border cells: edge-clamped (replicate), non-finite neighbours fall back to
  the centre value — the result is finite wherever the centre is finite.
  Side effect worth remembering for tests: on an inclined plane, edge
  columns/rows report a DIFFERENT (smaller) Horn slope than the interior, so
  slope-independence assertions must target interior cells.

### 2.3 Per-axis cell sizes — `src/terrain/ground/horizontalScale.ts`

`hornSlopeAspect(z, cols, rows, cellMetresX, cellMetresY)` takes **separate
X (east–west) and Y (north–south) cell sizes in metres**, produced by
`horizontalCellMetresXY(cellSizeM, isGeographic, latitudeDeg,
horizontalUnitToMetres)`:

- geographic frames: Y = cell × 111 320 m/°, X = Y × cos(latitude), latitude
  clamped to ±89°, non-finite latitude → cos = 1 (honest isotropic fallback);
- projected frames: both axes = cell × `horizontalUnitToMetres`
  (~0.3048 for feet), isotropic.

Callers must pass the WORLD latitude, not the render-recentred origin
(`cellConfidence.ts` documents this trap). **VRM/TPI must not re-derive any
of this: they take the already-computed slope/aspect grids (VRM) or work in
cell space (TPI radius is in cells), and where slope enters TPI's class
split (the 5° flat/middle test) they consume the same rise/run tangent.**

### 2.4 Aspect-convention mapping for VRM unit normals (Sappington 2007)

Sappington decomposes the per-cell unit surface normal as
`x = sinθ·sin(A), y = sinθ·cos(A), z = cosθ` with θ the slope ANGLE and A the
compass ASPECT AZIMUTH (clockwise from north, downslope). Adapting to OLV:

- θ from the tangent `m = slope` (rise/run): `sinθ = m/√(1+m²)`,
  `cosθ = 1/√(1+m²)` — no `atan` needed.
- OLV's `aspect` α is math-frame CCW-from-east downslope, so the unit
  downslope direction in (east, north) components is `(cos α, sin α)`, and:

  ```
  n_east  = sinθ · cos α        (= Sappington x = sinθ·sin A, A = 90° − α)
  n_north = sinθ · sin α        (= Sappington y = sinθ·cos A)
  n_up    = cosθ
  ```

  This is exactly the upward unit normal `(−p, −q, 1)/√(1+p²+q²)` for
  gradient `(p, q)`; the horizontal part points downslope with magnitude
  sinθ. The compass↔math conversion (`A = 90° − α`) only relabels the two
  horizontal components — the resultant length R, and therefore VRM = 1 − R/n,
  is identical. Flat cells (m = 0, sentinel α = 0) contribute (0, 0, 1)
  regardless of α.

## 3. How "Terrain Complexity" is derived today, and where it renders

- **Classifier:** `classifyComplexity()` in `src/terrain/datasetIntelligence.ts`
  buckets `meanSlopeDeg` / `meanRoughness` / `elevationVariance` (equal-weight,
  normalised 0..1 against 45° / 0.5 m RMS / 25 m²) into
  `ComplexityBucket = 'unknown' | 'low' | 'moderate' | 'high' | 'very-high'`.
  With NO engine signal it returns `'unknown'` → label "—" (the v0.3.10
  honesty pass: never fabricate a confident "Low").
- **Feeding path is header-derived:** `src/app/inspectorCardRefreshers.ts`
  pushes a load-time summary built from header facts alone (pointCount, bbox
  volume, source kind). It supplies NONE of `meanSlopeDeg/meanRoughness/
  elevationVariance`, so in practice the Complexity row reads "—" until a
  sampled engine pass provides those aggregates. There is currently **no
  raster-derived, literature-defined complexity metric anywhere** — that is
  the v0.5.4 gap TPI/VRM fill.
- **Render surfaces:**
  - `src/ui/DatasetIntelligenceCard.ts` — "Terrain Complexity" row
    (Low/Moderate/High/Very High); complexity is DESCRIPTIVE so its tier dot
    is always `neutral` (`signalTier('complexity', …)`) — a rugged site is
    not "worse" than a flat one. New metrics must keep that neutrality.
  - `src/terrain/export/terrainReportContent.ts` — report row
    `'Terrain complexity'` reuses the SAME summarised label (no re-derivation
    at export time, to avoid drift).
- **Vocabulary already in force:** buckets are lower-bound estimates when
  signals are missing; unknown renders as "—"; nothing invents a number.

## 4. TerrainContracts honesty envelope the new metrics must carry

From `src/terrain/TerrainContracts.ts` (`TerrainCoverageMeta` — "the honesty
fields every result MUST carry"):

- `coverage: TerrainCoverageMode` — `'full' | 'resident-only' | 'sampled'`.
- `sourcePointCount` / `analyzedPointCount` — declared vs actually walked.
- `confidence?: number` — 0..100, **optional and only set when derived from a
  real measurement** (the v0.3.10 pass removed hardcoded 60/50 constants;
  unset renders "—"). For the complexity cores this must be computed from
  data support (valid-cell/void fraction, window truncation at edges/voids),
  never asserted.
- `warnings: ReadonlyArray<string>` — ordered, human-readable caveats; the UI
  preserves order.

Additional established conventions the new results follow:

- Dispersion is mandatory: report median + IQR (project-wide type-7 quantile
  helper `src/terrain/quantile.ts` / `quantileSorted`), never a bare mean.
- Pure-data leaf rules: no DOM, no three.js, no I/O, deterministic — the
  `src/terrain/ground/*` headers are the style contract.
- New cores live in `src/terrain/complexity/` and are NOT imported by the
  app shell yet, so the `index` bundle must not grow
  (`scripts/check-bundle-budget.mjs` ceilings: index ≤ 760 KiB live).
- Test bucketing: `terrainComplexity.test.ts` lands in the `terrain` bucket
  (`scripts/test-bucket.mjs` TERRAIN regex matches `^terrain`);
  `reproPack.test.ts` stays in `unit`. Repro experiments register by adding
  a `test()` in `tests/reproPack.test.ts` that writes into the shared
  `metrics` record, which `afterAll` serialises to `benchmarks/out/
  metrics.json` + `metrics.md` under `REPRO=1`; `benchmarks/plots.py` reads
  only the keys it knows, so adding a new key is non-breaking.
