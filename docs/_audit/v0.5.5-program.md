## 1. Program purpose

v0.5.5 is a focused navigation, interaction, loading, and streaming-performance
release. Its centrepiece is direct manipulation: the user selects a hand tool,
grabs the whole scene, and drags it — smoothly, predictably, on mouse, trackpad,
pen, and touch — while large point clouds continue streaming underneath.

Around that centrepiece the release delivers: smoother wheel and trackpad zoom;
predictable orbit and pivot behavior; lower interaction frame time; faster
first useful view; coarse-first viewport coverage; center-first refinement;
camera-motion prediction; separated fetch/decode/GPU-upload pressure; accurate
loading-stage feedback; clean panel input ownership; and measurable regression
protection for the existing scheduler.

This is not a feature-expansion release. Correctness, local-first behavior,
memory safeguards, scientific honesty, and rendering quality established in
v0.5.4 are preserved throughout.

## 4. Program principles

**4.1 One control system.** Interaction state, FPS pressure, memory pressure,
LOD selection, DPR, request concurrency, and eviction compose into one
understandable system. No independent controllers that fight each other.

**4.2 Selection is not residency.** The selection budget controls how much
detail the scheduler actively selects; the residency budget controls the hard
memory ceiling and eviction. A temporary interaction-quality reduction must not
evict useful resident nodes or cause fetch-decode-upload churn.

**4.3 CSS pixels define data refinement.** LOD decisions use CSS viewport
dimensions, never framebuffer dimensions. Adaptive DPR must not change which
data resolution the scheduler considers necessary.

**4.4 Time-based behavior.** Damping, inertia, blending, and transitions use
elapsed seconds, not frame counts. The same synthetic input sequence behaves
consistently at 60, 120, and 144 Hz.

**4.5 Coarse coverage before decorative detail.** Avoid holes first, sharpen
the center second, refine the periphery last.

**4.6 Main-thread work is budgeted.** GPU preparation, uploads, and DOM
refreshes have explicit per-frame or per-interval budgets.

**4.7 Measured tuning.** Every numeric constant in this document is a starting
hypothesis, not a default, until measured on real desktop and mobile-class
devices.

### P1 — Hand tool: grab and drag the whole scene (flagship)

The user selects a hand icon, the cursor becomes an open hand, and a
primary-button drag grabs the entire scene and moves it with the pointer. This
must feel physically attached: the point under the cursor stays under the
cursor for the whole drag.

**Foundation (verified):** pan geometry already works — OrbitControls
right-drag pan and the custom two-finger touch pan (`Viewer.ts:1119-1130`).
The hand tool re-maps this capability onto primary drag with a first-class
tool affordance. It extends the existing architecture: `NavMode` gains
`'pan'` (`NavController.ts:37`), mode gating follows the existing
`_controls.enabled` pattern (`NavController.ts:230`), and the mode joins the
existing Digit1/2/3 switch group.

**Toolbar:** hand icon drawn in the dockIcons house style (24×24,
`currentColor`, 1.6 px stroke, round caps — `src/ui/dockIcons.ts`); tooltip
"Pan view"; stable label + `aria-pressed` per the toolDock pattern
(`toolDock.ts:164-168`); visible selected state; touch-friendly hit target;
listed in the Help overlay and first-run guidance. **Keyboard shortcut: `G`
(verified unbound; `H` is taken by the controls HUD — `NavController.ts:504`).
Re-verify against the full key map before landing.**

**Interaction model (hand tool active):**

- Primary-button drag pans camera and target parallel to the view plane; the
  grabbed point tracks the pointer 1:1.
- Wheel and trackpad continue to dolly (P2 controller).
- Pointer capture from drag start to release/cancel/window blur/mode change;
  dragging beyond the canvas edge stays stable.
- Text selection and page drag suppressed only while the viewer owns the
  gesture; panel scrolling unaffected.
- Touch: one-finger drag pans while the hand tool is active; two-finger pinch
  remains zoom; the existing twist/pinch/pan decomposition
  (`Viewer.ts:1119-1130`) remains the two-finger authority and its behavior is
  explicitly re-tested with the tool active.
- A mode change or tool-dock activation cancels any unfinished gesture safely;
  the hand tool can never start a measurement or annotation gesture.

**Pan geometry:**

1. At pointer-down, lock a world-space plane: normal along camera forward,
   through the current orbit target — or through a picked visible point under
   the cursor when picking is reliable.
2. Raycast the initial pointer onto the locked plane.
3. Raycast each subsequent pointer position onto the same plane.
4. Translate camera and target by the world-space delta.
5. Preserve camera-target distance and orientation exactly.
6. Never recompute a different plane mid-gesture (causes nonlinear drift).

**Screen-space fallback** when ray-plane intersection is unstable (grazing
angles): derive world-units-per-pixel at target distance; translate along
camera right/up; clamp motion when target distance approaches zero.

**Temporary grab, any mode:** middle-mouse drag pans regardless of the active
mode (verified unbound). Optionally Space-drag *in orbit mode only* — Space is
taken as "up" in walk/fly (`NavController.ts:534`), so any Space mapping must
be orbit-scoped and conflict-tested. Releasing the modifier restores the prior
mode.

**Cursors:** `grab` while the hand tool is selected and idle, `grabbing`
during drag — reusing the existing measure-handle pattern
(`style.css:2819-2820`); reset on pointer cancel and window blur.

**Inertia:** off by default. If added after measurement: velocity from recent
pointer samples, time-based exponential decay, world-velocity clamp, immediate
stop on new gesture, honors reduced motion, respects camera bounds, informs
scheduler prediction (P3).

**Precision:** pan translation operates in the viewer's working coordinate
frame. Before implementation, verify how large georeferenced coordinates are
recentred in the current pipeline and ensure per-frame pan deltas are never
accumulated into float32 world-scale coordinates.

**Tests:** mode selection; shortcut; `aria-pressed`; drag direction; camera
and target translate by the same vector; camera-target distance constant
within numerical tolerance; grabbed-point screen error within tolerance across
a scripted drag; pointer capture/cancel/blur; mode switch mid-drag; temporary
modifier restore; one-finger touch pan + two-finger zoom with the tool active;
panel wheel isolation; no accidental measurement; high-coordinate dataset;
during-active-streaming responsiveness.

**Acceptance:** grabbed point stays under the pointer within a small CSS-pixel
tolerance for the full drag; no jump at pointer-down; no drift after
pointer-up with inertia off; works with mouse, trackpad, pen, touch; stays
responsive while streaming; p95 pan frame time meets §7.

### P2 — Application-owned wheel/trackpad dolly controller

New lazy module `src/render/WheelDollyController.ts` (register in
`lazyChunks.ts`; add to the live-build exclude list — per-frame hot path).

**Ownership:** capture-phase wheel at the canvas; disable OrbitControls wheel
zoom only after the replacement passes parity tests; keep OrbitControls
orbit/pan; never touch wheel events over scrollable panels. Preserve the
existing zoom-to-cursor behavior (`Viewer.ts:1117`) — this controller owns
*when and how much*, not *where*.

**Delta normalization:** `DOM_DELTA_PIXEL` / `LINE` / `PAGE` → CSS pixels /
measured-or-configured line height / viewport height.

**Input classification:** distinguish trackpad streams from wheel notches by
multiple signals (event frequency, delta magnitude and continuity, fractional
deltas, burst duration); degrade safely when ambiguous.

**Velocity model** (log-space, time-based):

```ts
velocity += normalizedDelta * sensitivity;
const scale = Math.exp(velocity * deltaSeconds);
applyCursorCenteredDolly(scale);
velocity *= Math.exp(-frictionPerSecond * deltaSeconds);
```

Starting hypotheses (not defaults until benchmarked): trackpad sensitivity
0.0015–0.0025; wheel 0.08–0.12 per notch; friction 14–20 /s; max frame scale
≈1.15; settle |v| < 0.002.

**Stable cursor pivot:** pick a visible point under the cursor at gesture
start (or intersect a stable target plane); hold that pivot for the burst;
release/blend after settle; never pick per event if it causes readback stalls.

**Bounds:** clamp at min/max distance; zero out velocity pushing past a bound
(no hidden accumulation); allow immediate reversal.

**Reduced motion:** shorten or disable inertia; keep normalized direct
response. (This program introduces `prefers-reduced-motion` support — see §3.)

**Tests:** all three delta modes; notch stream; high-frequency stream;
mid-burst reversal; min/max distance; pivot stability; 60/120/144 Hz replay —
same synthetic sequence, same final radius within tolerance; background-tab
recovery; panel ownership; browser-zoom modifier; platform natural-scrolling
variants where hardware exists (document gaps honestly).

**Acceptance:** refresh-rate-independent final radius; cursor anchor within
~1–2 CSS px in the benchmark scene; no post-settle drift; no page scroll while
the canvas owns the gesture; no canvas zoom while a panel owns it.

### P3 — Motion model: linear, angular, zoom, tween destinations

Angular velocity via quaternion angular distance (never wrapped Euler
differences):

```ts
const dot = Math.abs(qC.x*qP.x + qC.y*qP.y + qC.z*qP.z + qC.w*qP.w);
const angle = 2 * Math.acos(Math.min(1, dot));
const angularVelocity = angle / deltaSeconds;
```

Track: linear velocity; angular velocity; zoom/dolly velocity; drag, wheel,
and hand-tool active states; current orbit pivot and target; tween destination
and estimated arrival (from `tweenTo`, `NavController.ts:278`). Predict
100–250 ms ahead; shrink the window under unstable frame timing, abrupt input
reversal, low-confidence estimates, or bounds collisions.

Tween-destination hints for: frame cloud, double-click focus, saved view,
fly-to annotation, reset view, share-link pose restore, hand-tool momentum
release (if inertia ships). Destination prefetch capped at 10–20% of request
capacity; current-view coverage always outranks it.

**Acceptance:** pure rotation triggers scheduler motion; a stationary camera
reads ~0 angular velocity; quaternion sign flips create no false rotation;
reversal invalidates stale prediction; visible coverage outranks prefetch.

### P4 — Pixel-space coverage and point-spacing refinement

Extend the scheduler input (`StreamingScheduler.ts:45-58`):

```ts
interface SchedulerView {
  viewProjection: ArrayLike<number>;
  cameraPosition: readonly [number, number, number];
  cameraDirection: readonly [number, number, number];
  cameraOrientation?: readonly [number, number, number, number];
  verticalFovRadians: number;
  viewportWidthCss: number;
  viewportHeightCss: number;
  frameTimeMs?: number;
}
```

Compute both quantities and keep them distinct (projected node diameter alone
is not a sufficient error metric):

```text
coveragePixels      = nodeWorldDiameter × viewportHeightCss / (2 × distance × tan(verticalFovRadians/2))
spacingErrorPixels  = node.spacing      × viewportHeightCss / (2 × distance × tan(verticalFovRadians/2))
```

`coveragePixels` estimates screen influence; `spacingErrorPixels` estimates
whether internal sample spacing is visibly coarse. Conservative near-distance
handling for nodes containing or intersecting the camera. A node is a
refinement candidate when projected spacing exceeds the current phase's pixel
target, children exist, regulators permit, and coarse-coverage invariants hold.

**Regression pin:** capture the v0.5.4 selected-node set for a reference
dataset and camera before switching the default. After: coarse coverage must
not regress; selection is explainable via projected spacing; FOV changes
produce sensible refinement changes; **DPR changes alone never alter
selection**; a 2× CSS viewport-height change approximately doubles projected
pixel values; memory/FPS regulators stay active. Ship behind
`?streamingScore=legacy` for A/B comparison.

### P5 — Single frame authority + adaptive rendering

**Fold the fade RAF into the main loop.** Remove the independent fade lifecycle
(`StreamingRenderer.ts:357-358`). The main viewer update becomes the sole
frame authority for: camera controls, wheel inertia, hand-tool inertia (if
shipped), scheduler phase, node fades, EDL motion state, adaptive DPR
transitions, GPU upload budget, render invalidation. No duplicate RAF may
survive viewer disposal.

**Adaptive DPR** — discrete phase states, never per-frame `setPixelRatio`:

```text
moving:   min(devicePixelRatio, 1.0)   // 1.10–1.15 if device measurement justifies
settling: min(devicePixelRatio, 1.25)
stable:   min(devicePixelRatio, 1.5)   // current MAX_PIXEL_RATIO
```

Transitions only at controlled state changes, with hysteresis and a minimum
interval; resize all dependent EDL targets coherently. 1.5 DPR shades 2.25×
the pixels of 1.0, but actual GPU savings must be measured (overdraw and
post-processing vary).

**Selection-budget multiplier** (selection budget only — never residency):
active drag/wheel/pan 0.55–0.70; settling 0.80; stable 1.00.

**Residency protection:** interaction alone never evicts. Eviction remains
governed by memory pressure, the hard ceiling, view obsolescence, and the
existing 2 s deferred-eviction policy.

**Acceptance:** one frame loop owns all transitions; no DPR oscillation;
interaction causes no evict-reload churn; quality returns without a visible
jump; existing EDL motion behavior remains correct.

### P6 — Refinement phases and foveated scheduling

```ts
type RefinementPhase = 'moving' | 'coverage' | 'center-refine' | 'full-refine';
```

**moving:** reduced selection budget, low DPR, preserve ancestors, request
missing coarse visible nodes, allow only high-value central or predicted nodes
beyond coverage, aggressively reject stale work, no peripheral speculation.
**coverage:** enter as motion slows; drive coarse visible coverage to a defined
threshold — prioritize as though the viewport were roughly 30% of its actual
height; keep parents until required children display; never transition on
elapsed time alone. **center-refine:** prioritize projected spacing error near
the center; limit the peripheral request share; keep honoring memory and FPS
pressure. **full-refine:** restore stable targets; refine the full viewport;
low-priority nearby prefetch only after visible goals are met. Reuse the
existing 250 ms settle window (`STABLE_SETTLE_MS`).

**Center weighting:** projected node position in NDC with bounding-box extent
(a large node crossing the center is not peripheral merely because its center
lies outside); a camera-forward dot product is only a cheap fallback:

```text
centerDistance = sqrt((projX/aspectWx)² + (projY/aspectWy)²)
centerWeight   = clamp(1 − centerDistance, 0, 1)
```

**Coverage metric** must not double-count overlapping ancestors and children —
choose, document, and test one of: visible screen-tile mask; conservative
projected rectangles; octree coverage accounting at a coarse depth; explicit
required-child readiness.

**Regulator composition — one function, consistent hysteresis:**

```text
effectiveDepthLimit      = min(base, velocityCap, fpsPressureCap, memoryPressureCap, phaseCap)
effectiveSelectionBudget = base × deviceFactor × fpsFactor × interactionFactor × phaseFactor
```

**Invariants:** missing coarse coverage outranks central refinement; parent
removal waits for displayed child coverage; phase changes cannot force
memory-budget oscillation; visible work outranks destination preload; central
work cannot starve holes; peripheral work cannot starve the center once
coverage is complete; stale generation results never become visible.

**Acceptance:** coarse view no later than the v0.5.4 baseline; central detail
measurably sooner; no new visible holes; no parent-child flicker; no request
surge after camera stop.

### P7 — Time-budgeted GPU upload queue

New lazy module `src/render/GpuUploadQueue.ts` (register in `lazyChunks.ts`;
live-build exclude list).

**Lifecycle:** `requested → fetching → fetched → decoding →
decoded-pending-upload → uploading → displayed | error | canceled`.

**Responsibilities:** accept decoded payloads; preserve dataset and generation
IDs; estimate payload bytes; reject stale items before upload; process under a
per-frame time budget (desktop ≈4 ms, mobile ≈2 ms, adaptive down under frame
pressure); bound decoded-pending-upload memory; expose queue depth and bytes;
support cancellation; dispose transferred resources safely.

**Granularity:** if a node upload cannot be split safely, cap complete uploads
per frame and measure worst-case node time. If chunked: preserve attribute
alignment; expose the node only when all required ranges are ready; never
display inconsistent attributes.

**Backpressure:** when pending-upload bytes exceed the limit — pause new
decodes; retain compressed chunks only within their own byte limit; prioritize
visible central nodes; cancel stale low-priority work.

**Loading UI:** extend the existing stage enum and labels
(`src/io/loadProgress.ts:15-39`) — the `uploading` stage ("Preparing GPU
buffers") already exists; surface the equivalent stages on the streaming path
and never report completion before the required data is *displayed*.

**Acceptance:** upload time visible in debug telemetry; bounded backlog; no
avoidable long main-thread stall on large-node completion; stale decodes
discarded before upload; "complete" means displayed.

### P8 — Fetch/decode pipeline separation (evidence-gated)

Proceed only if P0/P7 measurements show remaining value; narrow or defer if P7
resolves the observed stalls.

Limits: `maxConcurrentFetches`; max compressed queued bytes;
`maxConcurrentDecodes`; max decoded-pending-upload bytes; upload ms/frame.
Starting hypotheses — desktop 6 fetch / 3 decode; mobile 3 / 1–2; uploads
time-budgeted. Adaptation must be slow and hysteretic, driven by smoothed RTT
and throughput, failure rate, queue depth, decode utilization, and upload
backlog; never raise network concurrency while decode or upload backpressure
is high.

Abort semantics: `AbortController` for fetches; retain fetched compressed
chunks only if still useful and within budget; decode tasks check generation
IDs; stale completed decodes never upload; prompt reference release.

**Acceptance:** a slow CPU cannot accumulate unbounded fetched data; a slow
network can saturate decode capacity; byte limits enforced; cancellation and
retry stay correct.

### P9 — Panel scrolling and input ownership

CSS for scrollable panels: `overflow-y: auto; overscroll-behavior: contain;
scrollbar-gutter: stable; -webkit-overflow-scrolling: touch;`. For large
offscreen sections where measured useful: `content-visibility: auto;
contain-intrinsic-size: auto 400px;` — never applied blindly to controls
needing immediate measurement or focus.

Wheel ownership: panels stop wheel propagation with `{ passive: true }` (no
`preventDefault` on normal scrolling); the canvas controller additionally
verifies the event target belongs to the interactive viewport.

Active-scroll simplification only where profiling shows jank (reduce measured
expensive blur/shadow during scroll; restore after settle). Virtualize only
lists proven expensive; keyboard navigation and accessibility stay intact.

**Acceptance:** panel wheel never zooms the cloud; canvas wheel never scrolls
the page mid-gesture; touch stays natural; no focus regression; no speculative
virtualization complexity.

### P10 — Limited shader-filter foundation (conditional)

classification mask (`classMaskUniform.ts:16-31`, `Float32Array(256)`, shared
by both render backends):

```ts
interface PointFilterState {
  classificationMask?: Uint32Array;
  elevationRange?: readonly [number, number];
  intensityRange?: readonly [number, number];

```

Required: preserve classification filtering byte-for-byte; define generalized

disabled; memory accounting includes newly resident attributes.

## 6. Navigation state architecture

One navigation coordinator; no subsystem independently mutates the camera or
infers motion from unrelated signals. Extend the existing `NavController`
(`NavMode` gains `'pan'`; the `_controls.enabled` gating pattern at
`NavController.ts:230` extends naturally) rather than building a parallel
controller.

```ts
type InteractionState =
  | 'idle' | 'orbiting' | 'panning' | 'dollying'
  | 'walking' | 'flying' | 'tweening' | 'settling';
```

The coordinator emits ONE normalized motion snapshot per frame, consumed by
the scheduler, the DPR controller, the EDL gate, telemetry, predictive
loading, and cursor state. It exposes
`setMode / update(deltaSeconds) / cancelActiveGesture(reason) /
setReducedMotion / dispose`.

## 7. Performance targets (provisional until P0 baselines exist)

Navigation — desktop: p95 active-navigation frame time < 20 ms on the
reference device; mobile-class: < 33 ms. Plus: no avoidable main-thread task
> 50 ms during navigation; no camera jump on gesture start; no post-input
drift with inertia off; consistent behavior at 60/120/144 Hz; panel scroll
never triggers canvas interaction; the hand tool's grabbed point stays under
the pointer within tolerance.

Streaming — track time to first points / coarse coverage / central detail /
stable refinement; useful-byte ratio; stale canceled and decoded bytes; upload
backlog; resident-node churn. Required: coarse coverage does not materially
regress; central detail measurably improves; cancellation waste does not rise
without justification; interaction causes no evict-reload cycles.

GPU upload — budget respected in normal cases; bounded backlog; stale uploads
rejected; completion UI corresponds to displayed state.

## 16. Risk register

- **Competing adaptive controllers** → centralized effective-budget function,
  single interaction snapshot, explicit hysteresis, regression pins.
- **Bundle budget exhaustion (~5 KiB index headroom)** → all new modules lazy
  (`lazyChunks.ts` registration), `lint:main-deferral`, per-PR `check:bundle`;
  any index growth is a documented maintainer decision.
- **Live-build slowdown in new hot paths** → exclude-list additions in
  `vite.config.ts:62-88` for the hand tool, wheel controller, upload queue,
  and scoring changes; live-build spot benchmarks before performance claims.
- **Hand-tool direction feels reversed** → define the visual convention
  ("grab the scene": scene follows the pointer), user-test, keep the math
  consistent everywhere.
- **Cursor picking stalls navigation** → pick only at gesture start, cache the
  pivot, plane fallback, profile readback.
- **Adaptive DPR resize jank** → discrete states, minimum interval, no
  per-frame setPixelRatio, resize all targets together.
- **Lower interaction budget causes reload churn** → selection ≠ residency,
  preserve deferred eviction, measure node churn.
- **Upload queue grows memory** → byte-based backlog limit, decode
  backpressure, stale rejection, telemetry.
- **Pixel-space scoring over-refines** → spacing-based thresholds, phase
  targets, memory/FPS caps, legacy A/B flag.
- **Touch gesture conflicts** → the existing two-finger decomposition stays
  authoritative; explicit gesture table; capture tests; device matrix; no
  ambiguous tool state.
- **Scope expansion** → full filter suite deferred, pipeline split
  evidence-gated, every optional item has a cut line.
- **Left rail hides controls or breaks mobile (P11)** → desktop/tablet-only gate
  (`MobileSheet` owns mobile untouched); a persistent reopen grabber tab so the rail is
  never fully lost; measured `--olv-dock-clear` with a safe fallback constant; a
  bounding-rect test that actionable panel controls always sit above the dock;
  reduced-motion path; class names preserved so existing selectors/tests resolve.

## 18. Definition of done

A user can: select the hand icon and drag the whole scene smoothly and

central detail sharpen first; stop moving without an upload freeze; scroll
panels without zooming the canvas; and always know whether data is
downloading, decoding, uploading, displayed, or refining.

controller; inspect scheduler phase and queue pressure; verify zero residency
churn from interaction; confirm every pre-existing correctness regulator is
still active; and release v0.5.5 with complete evidence and honest

## Final program statement

v0.5.5 strengthens OpenLiDARViewer as a native, high-precision LiDAR
inspection environment by improving the whole interaction loop — input to
camera, camera to scheduler, scheduler to decoder, decoder to GPU, GPU to the
user's eye. The hand tool is the heart of that contract: choose the hand, grab
the scene, and drag it directly — and stay in control while the data keeps
streaming.
